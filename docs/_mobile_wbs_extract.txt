################ ID 4.1 | Authentication & Dashboard > Authentication & Dashboard ################

################ ID 4.1.1 | Authentication & Dashboard > Login ################
--- DESCRIPTION/ASSUMPTIONS ---
- Accessible by Building Manager on mobile (Android & iOS).
- Account created by Super Admin or Building Manager via web portal.
- BM does NOT belong to a User Group. Scope defined by buildings assigned to BM account.
- On successful login => Dashboard (4.1.5) scoped to BM's assigned buildings.
- If a valid session already exists => system skips Login and navigates directly to Dashboard.
- Maximum 5 consecutive failed attempts before 15-minute lockout.
- All connections use HTTPS/SSL with TLS 1.2+.
--- FIELDS/COMPONENTS ---
Screen: Login (Building Manager Mobile)

- EZAxis logo - Display - Centered top
- Email - Text field - Required - Keyboard type: email - Placeholder: "Enter your email" - Max 100 characters
- Password - Text field (masked) - Required - Placeholder: "Enter your password" - Min 8 characters - Show/hide password toggle icon
- Forgot Password? - Hyperlink - Below password field - Navigates to Forgot Password (4.1.3)
- Login - Button - Primary - Full-width - Large tap target
- Loading indicator while authenticating
--- USER FLOW ---
Screen: Login (Building Manager Mobile)

Preconditions:
- No authentication required - entry point for unauthenticated BM on mobile app.
- BM account must exist with role = Building Manager.
- If valid session token is stored on device => system skips Login and navigates directly to Dashboard.

1. App opens. System checks for stored session token on device:
   1.1. Valid session token found => navigates directly to Dashboard (4.1.5) => flow ends.
   1.2. No token or expired token => renders Login screen.

2. BM enters email and password => taps [Login].
   + Keyboard dismisses automatically when [Login] tapped.
   + Loading spinner shown on [Login] button while authenticating.

3. System validates fields:
   3.1. Email empty => inline error below field: "Email is required." => does NOT authenticate.
   3.2. Password empty => inline error: "Password is required." => does NOT authenticate.
   3.3. Email format invalid => inline error: "Please enter a valid email address." => does NOT authenticate.

4. System checks failed attempt counter:
   4.1. >= 5 consecutive failed attempts within last 15 min => does NOT authenticate => error toast: "Too many failed attempts. Please try again in 15 minutes."

5. System queries backend:
   5.1. Email not found or role != Building Manager => does NOT create session => error toast: "Invalid email or password." => increments counter.
   5.2. Account Status = Inactive => does NOT create session => error: "Your account is not active. Please contact the administrator."
   5.3. Account Status = Pending (invitation not yet accepted) => does NOT create session => error: "Your account is not yet activated. Please check your email for the invitation link."
   5.4. Account Status = Inactive => error toast: "Your account is not active. Please contact the administrator."
   5.3. Password mismatch => does NOT create session => error toast: "Invalid email or password." => increments counter.
   5.4. All valid:
    + Session token stored securely on device (Keychain on iOS / Keystore on Android).
    + Failed attempt counter reset to 0.
    + Last login timestamp updated.
    + Navigates to Dashboard (4.1.5) scoped to BM's assigned buildings.

Edge cases:
- Network unavailable when tapping [Login] => error toast: "No internet connection. Please check your network and try again."
- BM's device is offline but has a valid cached session => app opens to last cached state (offline mode) if offline caching is supported; otherwise redirects to Login with message: "You are offline. Please connect to log in."
- BM pastes email with leading/trailing spaces => system trims whitespace before validation.
- BM has no assigned buildings (account misconfiguration) => login succeeds but Dashboard shows: "No buildings assigned to your account. Please contact the administrator."

################ ID 4.1.2 | (cont) > Logout ################
--- DESCRIPTION/ASSUMPTIONS ---
- Accessible by authenticated Building Manager from any screen in the mobile app.
- Logout accessible from profile/settings menu.
- Confirmation prompt shown before executing logout.
- On logout: session token immediately invalidated server-side and removed from device.
--- FIELDS/COMPONENTS ---
Component: Logout (Building Manager Mobile)

Access path: Bottom navigation => Profile/More => Sign out

- Sign out - Menu item - Triggers logout confirmation

Alert: Confirm Sign Out
- Message: "Are you sure you want to sign out?"
- Sign out - Button - Destructive style
- Cancel - Button - Default style
--- USER FLOW ---
Flow: Logout (Building Manager Mobile)

Preconditions:
- BM authenticated on mobile app.

1. BM taps Profile or More icon in bottom navigation.
2. Profile/Settings screen opens. BM taps [Sign out] menu item.
3. System shows native OS alert dialog:
   + Title: "Sign Out"
   + Message: "Are you sure you want to sign out?"
   + Buttons: [Sign out] (destructive style) | [Cancel].
4a. BM taps [Sign out]:
   + Session token invalidated server-side immediately.
   + Session token removed from device secure storage (Keychain/Keystore).
   + Any cached data cleared from app local storage.
   + App navigates to Login screen (4.1.1).
4b. BM taps [Cancel]:
   + Alert dismisses.
   + BM remains on Profile/Settings screen, fully authenticated.

Edge cases:
- Network unavailable when logout tapped => system still removes local token and navigates to Login. Server-side invalidation queued and retried when online (or token expires naturally server-side).
- BM's session expires naturally while app is in background => on next foreground action requiring API, system detects 401 response => clears local token => navigates to Login with toast: "Your session has expired. Please log in again."
- App force-closed and reopened => system re-checks stored token on app open (step 1 of Login flow).

################ ID 4.1.3 | (cont) > Forgot Password ################
--- DESCRIPTION/ASSUMPTIONS ---
- Accessible by unauthenticated Building Managers from the Login screen.
- Same OTP-based 2-screen flow as web (4.1.3 mirrors 3.1.3 logic).
- OTP sent to registered email via AWS SES/SendGrid.
- OTP validity: 10 minutes (assumed - flagged for BA review).
- After reset => navigates to Login screen (4.1.1).
--- FIELDS/COMPONENTS ---
Screen 1 - Enter Email (Mobile)

- Page title: "Forgot Password"
- Instruction text: "Enter your registered email address. We will send you a verification code."
- Email - Text field - Required - Keyboard type: email - Placeholder: "Enter your email"
- Continue - Button - Primary - Full-width
- Back to Login - Hyperlink

Screen 2 - Enter OTP and New Password (Mobile)

- Page title: "Reset Password"
- Instruction text: "Enter the verification code sent to [email]."
- OTP Code - Text field - Required - Keyboard type: numeric - 6 digits - Large input
- Resend Code - Hyperlink - Active after 60 seconds - Shows countdown timer
- New Password - Text field (masked) - Required - Min 8 characters - Show/hide toggle
- Confirm New Password - Text field (masked) - Required - Must match - Show/hide toggle
- Reset Password - Button - Primary - Full-width
- Back to Login - Hyperlink
--- USER FLOW ---
Screen: Forgot Password (Building Manager Mobile)

Preconditions:
- No authentication required.
- BM must have an active account with the provided email.

Screen 1 - Enter Email:
1. BM taps [Forgot Password?] on Login screen => Screen 1 loads.
2. BM enters email (keyboard type: email) => taps [Continue]:
   2.1. Empty => error: "Email is required."
   2.2. Format invalid => error: "Please enter a valid email address."
   2.3. Email not found in BM accounts => does NOT send OTP => error: "No account found with this email address."
   2.4. Account Inactive => does NOT send OTP => error: "Your account is not active. Please contact the administrator."
   2.5. Network unavailable => error: "No internet connection. Please check your network and try again."
   2.6. Valid => OTP sent via AWS SES/SendGrid => navigates to Screen 2.
3. BM taps [Back to Login] => navigates to Login (4.1.1). No OTP sent.

Screen 2 - Enter OTP & New Password:
4. Screen shows masked email OTP was sent to.
5. BM enters OTP (numeric keyboard, 6 digits) / New Password / Confirm:
   5.1. OTP empty => error: "Please enter the verification code."
   5.2. OTP wrong format => error: "Verification code must be 6 digits."
   5.3. OTP incorrect => error: "Invalid verification code. Please try again." After 5 wrong attempts => OTP invalidated => must restart from Screen 1.
   5.4. OTP expired (> 10 min) => error: "This code has expired. Please request a new one." => returns to Screen 1.
   5.5. Password < 8 chars => error: "Password must be at least 8 characters."
   5.6. Confirm mismatch => error: "Passwords do not match."
   5.7. Network unavailable => error: "No internet connection."
   5.8. All valid:
    + Password updated.
    + All active sessions (web + other mobile devices) invalidated.
    + Navigates to Login (4.1.1).
    + Success toast: "Password reset successfully. Please log in with your new password."
6. [Resend Code] (active after 60s with countdown timer):
   + New OTP sent => previous OTP invalidated => countdown resets.

Edge cases:
- Email delivery fails => "Failed to send verification code. Please try again later."
- BM closes app mid-reset and reopens => returns to Login screen (Screen 2 state not persisted).

################ ID 4.1.4 | (cont) > Multi-Language Switch ################
--- DESCRIPTION/ASSUMPTIONS ---
- Accessible by authenticated Building Manager from any screen.
- Supports English (EN) and Vietnamese (VI).
- Language switch applies immediately to all UI elements without app restart.
- Selected language persisted per user account across sessions and devices.
--- FIELDS/COMPONENTS ---
Component: Language Switcher (Building Manager Mobile)

Access path: Profile / Settings => Language

- Language - Selection list or toggle
- Options: English (EN) | Ti?ng Vi?t (VI)
- Currently selected language highlighted
- Tap to change - applies immediately
--- USER FLOW ---
Flow: Multi-Language Switch (Building Manager Mobile)

Preconditions:
- BM authenticated.
- Language setting accessible from Profile/Settings.

1. BM navigates to Profile => Settings => Language.
2. Current language highlighted in a selection list.
3. BM taps alternate language (EN or VI):
   3.1. App immediately re-renders all UI text in selected language without restart.
   3.2. Data content (asset names, descriptions) is NOT translated.
   3.3. System saves preference to BM's account via API.
   3.4. Save succeeds => preference persists across sessions and devices (web portal also reflects change).
   3.5. Save fails (network issue) => UI still shows new language for current session => error toast: "Failed to save language preference. The change will apply to this session only." => reverts to previous on next login.

Edge cases:
- BM switches language while a form is open (e.g., during Create WO) => field values preserved; labels/placeholders update.
- BM switches language while offline => UI changes locally but server sync fails => preference reverts on next login.

--- USER FLOW ---
System Process: Push Notification (Building Manager Mobile)

This is a system-triggered process, not a user-initiated flow.

--- Delivery mechanism ---

1. System event occurs that triggers a notification for BM.
2. Backend sends push notification payload to Firebase Cloud Messaging (FCM).
3. FCM routes the notification to all registered BM devices for this account.
4. Device is online => notification delivered immediately to device.
5. Device is offline => FCM queues the notification for delivery when device comes online. FCM message delivery is best-effort; messages may expire after 28 days if device remains offline.

--- Trigger events and notification content ---

Events that trigger BM mobile notifications:
+ New Tenant Request received for BM's building:
  Title: "New Service Request"
  Body: "A tenant has submitted a service request for [Building Name]. Tap to review."
  Action: opens Request Detail on tap.

+ New Ad-hoc WO Request pending BM approval (Supervisor submitted):
  Title: "Work Order Awaiting Approval"
  Body: "An ad-hoc work order request for [Building Name] requires your approval."
  Action: opens Request Detail.

+ WO ready for BM sign-off (status = Verified):
  Title: "Work Order Ready for Sign-off"
  Body: "Work Order [ID] has been reviewed by the Supervisor and is ready for your final sign-off."
  Action: opens WO Detail.

+ Spare part request awaiting BM approval (On-Hold or Unavailable):
  Title: "Spare Part Request"
  Body: "A spare part request for Work Order [ID] requires your approval."
  Action: opens WO Detail (spare parts tab).

+ Low-stock alert for spare part in BM's building:
  Title: "Low Stock Alert"
  Body: "Spare part [Name] stock is below minimum threshold. Current stock: [N]."
  Action: opens Spare Part Detail.

+ Maintenance Plan submitted by Supervisor (Pending approval):
  Title: "Maintenance Plan Pending Approval"
  Body: "A maintenance plan requires your approval for [Building Name]."
  Action: opens Plan Detail.

--- In-app notification handling ---

6. When app is open (foreground): notification displayed as in-app banner at top of screen. Auto-dismisses after 4 seconds. Tap to navigate.

7. When app is in background: OS displays notification in system tray. Tap to open app and navigate to relevant screen. Badge count increments on app icon.

8. When app is closed: OS displays notification. Tap to open app (cold start) and navigate to relevant detail screen.

9. BM taps notification from any state:
   + App opens (or comes to foreground).
   + Navigates directly to the relevant detail screen (Request Detail, WO Detail, Spare Part Detail, or Plan Detail).
   + Notification marked as read.
   + Badge count decrements.

10. BM has unread notifications: notification bell icon in app header shows badge count.

11. Notification permissions:
    + On first app install: system requests notification permission.
    + If BM denies => push notifications not delivered. In-app prompts user to enable in device settings.
    + BM can manage notification settings in device Settings => App => Notifications.

Edge cases:
- FCM token expires or device registration changes (e.g., app reinstall) => system refreshes FCM token on next app open and re-registers with backend.
- BM receives notification for a request that has already been processed by another session => tapping notification navigates to Detail screen which shows the current (already-processed) status.
- Multiple notifications received while offline => all queued and delivered in batch when device comes online. App processes them sequentially.
- BM logs out => FCM token unregistered => no more notifications until next login.

Edge cases:
- Building Technician not available on-site => pending client confirmation on how to handle. Flagged for BA review.

- Technician accidentaly taps wrong button on sign-off screen => no undo after each individual sign-off step is confirmed (pending BA confirmation).
- Sign-off method ASSUMED = Button confirmation for all 3 parties. No signature drawing or OTP required in MVP. Each party physically taps [Sign Off] / [Confirm] on the Technician's device.
